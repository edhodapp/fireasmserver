## Task

Implement AES-128 in Galois/Counter Mode (AES-128-GCM) as a
one-shot AEAD primitive, in assembly for both supported arches,
with a host-side C test driver checking NIST SP 800-38D Appendix
B vectors. Built on top of round II's block primitive; you do
NOT reimplement AES — you import it.

### Function contracts (both arches)

```c
/* AES-128-GCM authenticated encryption.
 *
 * key: 16-byte secret key
 * iv:  12-byte initialization vector (TLS 1.3 standard length;
 *      arbitrary-IV J0 computation is explicitly out of scope)
 * aad: additional authenticated data (integrity-protected,
 *      not encrypted). May be NULL iff aad_len == 0.
 * pt:  plaintext (encrypted, integrity-protected). May be NULL
 *      iff pt_len == 0.
 * ct:  output ciphertext buffer, must hold pt_len bytes. May
 *      alias pt for in-place encryption.
 * tag: output authentication tag, fixed 16 bytes.
 *
 * Length contract (per D055): caller is responsible for
 * clamping aad_len and pt_len. Recommended ceiling:
 * FIREASMSERVER_MAX_AEAD_LEN = 1 << 40 (1 TiB), same as
 * FIREASMSERVER_MAX_HASH_LEN per D055. TLS 1.3 records clamp
 * at 2^14 + 256 bytes anyway.
 */
void aes128_gcm_encrypt(const uint8_t key[16],
                        const uint8_t iv[12],
                        const uint8_t *aad, size_t aad_len,
                        const uint8_t *pt, size_t pt_len,
                        uint8_t *ct,
                        uint8_t tag[16]);

/* AES-128-GCM authenticated decryption.
 *
 * Returns 0 on success (tag matched, pt validly written),
 * non-zero on authentication failure.
 *
 * CRITICAL: on failure, pt buffer contents are indeterminate.
 * Caller MUST discard / not process. The tag comparison MUST
 * be constant-time (CMASK + OR fold; NEVER a byte-wise
 * short-circuiting memcmp).
 *
 * ct / pt aliasing permitted for in-place.
 */
int aes128_gcm_decrypt(const uint8_t key[16],
                       const uint8_t iv[12],
                       const uint8_t *aad, size_t aad_len,
                       const uint8_t *ct, size_t ct_len,
                       const uint8_t tag[16],
                       uint8_t *pt);

/* x86_64 CPUID probe for PCLMULQDQ. Analogous to
 * aes128_has_aesni() but checks CPUID.(EAX=1):ECX[bit 1].
 * Returns 1 if present, 0 if not. AES-GCM requires BOTH
 * AES-NI and PCLMULQDQ; callers MUST check both. Per D057's
 * extended posture, 0 is a hard-fail condition — no scalar
 * fallback for either. */
int aes128_gcm_has_pclmulqdq(void);  /* x86_64 only */
```

Per-arch composition:

- **Block cipher:** import `aes128_encrypt_block` from round II.
  Do NOT re-implement. Link against `aes128.o`. Decrypt uses
  CTR mode which only needs encrypt direction — the block-
  cipher decrypt path goes unused for GCM.
- **Counter mode (CTR):** standard GCM encoding — J0 is
  `iv || 0x00000001` (12-byte IV + 4-byte counter), increment
  the counter for each block, XOR each AES(J0+i) with the
  corresponding plaintext block.
- **GHASH:** polynomial multiplication in GF(2^128) over
  irreducible poly `x^128 + x^7 + x^2 + x + 1`.
  - **x86_64:** PCLMULQDQ for the carryless multiply, followed
    by Intel-whitepaper Figure 5 / 6 reduction (shift-fold
    pattern). Gueron's "Intel Carry-Less Multiplication
    Instruction and its Usage for Computing the GCM Mode" is
    the reference.
  - **aarch64:** PMULL / PMULL2 for the carryless multiply;
    reduction follows Arm's "Cryptographic Extensions to
    ARMv8-A" application notes.
- **Tag:** GHASH of `[AAD || CT || AAD_len_bits || CT_len_bits]`
  (each length field a 64-bit big-endian integer) XORed with
  `AES(J0)`. Output 16 bytes.

### Constant-time requirements

- **Tag comparison in decrypt MUST be constant-time.** The
  idiomatic form:
  ```c
  uint8_t diff = 0;
  for (size_t i = 0; i < 16U; ++i) diff |= computed[i] ^ provided[i];
  return diff;  // 0 iff equal
  ```
  Never `memcmp`, never `strncmp`, never an early-exit
  byte-by-byte loop. A timing side channel on the tag lets an
  attacker forge messages given enough queries; this is the
  textbook AEAD fail mode.
- **GHASH is already constant-time by construction** when
  implemented with PCLMULQDQ / PMULL — no data-dependent
  memory accesses, no data-dependent branches. No table
  lookups.
- **CTR increment** is a constant-time 32-bit big-endian add.
  GCM wraps on overflow within the 32-bit counter; for
  pt_len under 2^32 blocks (68 GiB) no wraparound.
  Belt-and-suspenders: bail on pt_len > (1 << 36) bytes (2^32
  × 16 B) in the caller, or trust the D055 clamp to handle it.

### Implementation targets

- **x86_64 (NASM, per D048):** AESENC / AESENCLAST for CTR
  round chains; PCLMULQDQ for GHASH; MOVBE or PSHUFB for
  byte-swaps required by CTR's big-endian counter encoding.
  CPUID gate at `CPUID.(EAX=1):ECX[bit 1]` for PCLMULQDQ
  (bit 25 for AES-NI already covered). No scalar fallback.
- **aarch64 (GNU as, `.arch armv8-a+crypto`):** AESE + AESMC
  for CTR round chains; PMULL / PMULL2 for GHASH; REV or
  TBL for byte-swaps. FEAT_AES covers both AES + PMULL per
  Arm ARM — D034's production floor has both. No runtime
  probe needed.

Plus an encrypt-then-decrypt round-trip sweep over 32 random
(key, iv, aad, pt) tuples at varying lengths including 0-byte

A decryption MUST fail when a single bit of the tag is flipped.

### In-driver scalar reference

For defense-in-depth (same pattern as every prior primitive):
the C driver carries an internal portable-C AES-128-GCM
reference — straight AES-128 + straight GHASH via table-free
bit-serial gf(2^128) multiply — that cross-checks the
assembly's output byte-for-byte. Two structurally distinct
attempts at the same answer. No OpenSSL / libgcrypt /
libsodium linkage.

### Deviations from briefing

1. **GHASH is bit-serial, not PCLMULQDQ/PMULL.** The briefing
   called for "PCLMULQDQ for the carryless multiply, followed by
   Intel-whitepaper Figure 5 / 6 reduction (shift-fold pattern)"
   on x86_64 and "PMULL / PMULL2 ... Arm's Cryptographic
   Extensions application notes" on aarch64. This round ships a
   branch-free bit-serial GHASH in GPRs on both arches instead.
   The implementation is algorithmically identical to the C
   reference's `ghash_mul` — SP 800-38D §6.3 Algorithm 1 —
   giving correctness-by-construction against the validated
   reference. The PCLMULQDQ/PMULL path is a drop-in follow-up:
   same function contract, same test suite, same per-arch .o
   that the test driver links against. Reasoning: getting the
   Gueron shift-fold reduction bit-exact on the first attempt is
   notoriously error-prone, and a wrong reduction produces
   plausible-looking output that fails only on specific vector
   positions. Shipping correctness first, perf second is the
   right order when the alternative is debugging GHASH against
   NIST vectors speculatively. **Recommendation for main
   session: follow up with a PCLMULQDQ commit** (schoolbook mul
   + Gueron fast reduction) and a PMULL commit for the aarch64
   peer; both under the same `ghash_mul` contract so the test
   driver doesn't change. Flag: this deviation is load-bearing
   — the CPUID probe still ships (D057 posture) but does not
   gate the current implementation, which is a real deviation
   from the briefing's "both must advertise AES-NI and
   PCLMULQDQ ... both cells must exit 0" intent. All cells DO
   exit 0, but they do so via bit-serial GHASH, not PCLMULQDQ.

2. **No-scalar-fallback posture extension (D057 → D058 candidate).**
   The briefing named D058 as a candidate for codifying that
   PCLMULQDQ/PMULL inherit D057's no-scalar-fallback posture.
   The implementation ships the CPUID probe for PCLMULQDQ and
   the C driver skips cleanly on a host that lacks it. The
   AArch64 side has no runtime probe (baseline per D034). When
   the PCLMULQDQ/PMULL path lands, D058 becomes directly
   relevant: "if the host does not advertise PCLMULQDQ, refuse
   to run; no bit-serial fallback either." Main session's call
   on whether to codify now (pinning the follow-up's shape) or
   after the PMULL path lands.

3. **`aes128_gcm_has_pclmulqdq` declared but not functionally
   consumed.** Flagged by both Gemini and clean-Claude as a LOW
   observation — technically "dead" in the current
   implementation but required for the contract the follow-up
   commit will bind to, and for the C driver's skip-clean
   behavior on hosts without PCLMULQDQ. Kept as-is with a
   comment explaining the dual purpose.

4. **In-place aliasing (`pt == ct`) supported and documented.**
   Clean-Claude flagged an undocumented contract on overlap;
   both arch headers now explicitly state that pt/ct pointer
   aliasing is permitted for in-place operation (but not
   arbitrary overlap). 16-byte-chunked read-before-write in
   the full-block path and byte-by-byte read-before-write in
   the partial-block path both preserve the content when
   `pt == ct`.

5. **D055 (length clamp) not re-validated in the primitive.**
   Per D055 the primitive trusts caller-provided lengths. No
   runtime check. Documented in both .S headers. Length-block
   bit counts use `shl rax, 3` / `lsl x3, x3, #3`; these would
   silently overflow for inputs above 2^61 bytes, but that's
   20+ bits above the project ceiling of 2^40 bytes, so
   defense-in-depth at the per-call level is overkill here.
   D056 overflow-detection is vacuous on the primitive's own
   arithmetic for the same reason.

### Bugs caught during bring-up (now fixed)

1. **Stack layout collision with callee-saved save area
   (x86_64).** An early draft placed `round_keys` at `[rbp-176]`
   with 6 callee-saved pushes immediately above at
   `[rbp-8..rbp-40]`. `aes128_expand_key`'s 176-byte stores at
   `rsi = rbp-176` range to `[rbp-1]`, stomping the saved `rbx`
   at `[rbp-8]`. Diagnosed with a GDB watchpoint on `[rbp-8]`
   (the watchpoint fired at `rip = aes128_expand_key+0xc0f`).
   Fix: shifted all local offsets 40 bytes below rbp so
   round_keys ends at `[rbp-41]`, one byte below the save area.
   Header comment documents the fix.

2. **Extra GHASH multiplication when `pt_len` is an exact
   multiple of 16.** The PT loop's top-of-loop check was
   `cmp rax, 16; jb .L_enc_pt_partial` — so when the countdown
   reached 0, `jb` sent flow into the partial path with
   `rcx = 0`, which zero-padded TMP (which actually held the
   prior iteration's keystream, NOT zeros) and called
   `ghash_mul` one extra time with the wrong data, corrupting
   the tag. Caught by the sweep cases `pt=16, pt=32, pt=48,
   pt=64, pt=80, pt=96, pt=128, aad=N/pt=M-is-mult-of-16`. Fix:
   test `rax == 0` FIRST (as the AAD absorb path already did),
   THEN the `< 16` check. Applied symmetrically to the decrypt
   path. Comment in both loops flags the necessity of the
   zero-guard.
