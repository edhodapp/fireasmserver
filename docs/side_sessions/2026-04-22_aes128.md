## Task

Implement AES-128 block encrypt + decrypt + key schedule,
conforming to FIPS PUB 197, in assembly for both supported
arches, with a host-side C test driver checking published
vectors. Raw block primitive only — ECB is not a mode we ship,
it's just the test harness for one-block-at-a-time correctness.
GCM mode is the follow-up session; do not anticipate it.

### Function contracts (both arches)

Host-callable leaf routines under SysV AMD64 / AAPCS64. Signature
spellings are C; the asm matches the ABI for direct C linkage.

```c
/* Expand a 128-bit key into 11 round keys (176 bytes).
 * Caller-provided output buffer must be 16-byte-aligned.
 * Used by both encrypt and decrypt paths — the decrypt path
 * applies aesimc (x86_64) or aesimc (aarch64) to the inner
 * round keys at decrypt time, NOT during expansion. That
 * choice trades a couple of extra aesimc instructions per
 * decrypt for a simpler, shared key schedule. */
void aes128_expand_key(const uint8_t key[16], uint8_t round_keys[176]);

/* One-block encrypt: round_keys from aes128_expand_key,
 * in and out 16-byte blocks (may alias). */
void aes128_encrypt_block(const uint8_t round_keys[176],
                          const uint8_t in[16],
                          uint8_t out[16]);

/* One-block decrypt: round_keys from the same expand_key,
 * in and out 16-byte blocks (may alias). The decrypt path
 * applies aesimc to rounds 1-9 at invocation time, so the
 * round_keys buffer is unchanged by decrypt. */
void aes128_decrypt_block(const uint8_t round_keys[176],
                          const uint8_t in[16],
                          uint8_t out[16]);

/* Public CPUID/HWCAP probe for x86_64. Returns 1 if AES-NI
 * is available, 0 otherwise. Caller uses this to skip the
 * runtime cell gracefully on non-AES-NI hardware. AArch64
 * has no counterpart — FEAT_AES is required by D034. */
int aes128_has_aesni(void);  /* x86_64 only */
```

- **Byte order:** input/output blocks are the FIPS 197 big-endian
  convention (first byte of the input array is the high byte of
  the block). AES-NI's `aesenc`/`aesdec` instructions operate on
  little-endian xmm registers; the asm handles the byte-swap on
  load/store without committing the caller to it.
- **Alias safety:** `in == out` is permitted. Intermediate state
  lives in registers; no write-through-buffer hazard.
- **No streaming API.** Callers compose ECB/CBC/CTR/GCM over top
  of `aes128_encrypt_block`. No modes in this primitive.

### Implementation targets

- **x86_64 (NASM, per D048):** use AES-NI instructions
  (`aeskeygenassist`, `aesenc`, `aesenclast`, `aesdec`,
  `aesdeclast`, `aesimc`). CPUID gate at
  **`CPUID.(EAX=1):ECX[bit 25]`** (verify against Intel SDM
  Vol. 2A before committing — the SHA-256 briefing had that
  typo and we don't repeat it). Key schedule follows Intel's
  white paper "Intel Advanced Encryption Standard (AES)
  Instructions Set" using `aeskeygenassist`; loop unroll all
  10 rounds rather than a counted loop. **No scalar fallback**
  — D034's x86_64 profiles target CPUs with AES-NI (Westmere+
  server, Goldmont+ Atom). If `aes128_has_aesni()` returns 0,
  fail hard rather than fall back to a table-lookup path that
  would violate constant-time. Document the
  "no-fallback is the discipline choice" decision in the
  deviations section of this briefing's Status.

- **AArch64 (GNU as, `.arch armv8-a+crypto`, per D034):** use
  FEAT_AES (`aese`, `aesd`, `aesmc`, `aesimc`). No hardware
  key-expansion instruction — do the key schedule in plain
  SIMD, following the algorithm in FIPS 197 §5.2 (`RotWord`,
  `SubWord`, `Rcon` XOR). No runtime CPU dispatch — FEAT_AES
  is required on every D034 production profile (Cortex-A76+,
  Neoverse N1/V1/V2, Apple cores); Cortex-A72 / A55 are
  explicitly out of scope.

- **Both arches: constant-time by construction.** AES-NI and
  FEAT_AES hardware instructions are data-independent in timing
  per Intel SDM / ARM ARM. No data-dependent branches; no
  data-dependent memory accesses (round keys are accessed
  sequentially). The key schedule's only branch is the loop
  counter, which depends on the literal round number 0..10, not
  the key.

- **D056 application:** the C driver's length math lives at the
  trust boundary. Each `len` / offset in the driver loops uses
  `size_t` and the compile-as-lint catches signed-overflow traps.
  In the asm itself there's no external-input arithmetic — the
  primitive takes fixed 16-byte blocks and a fixed 176-byte
  round-key buffer.

- **D055 application:** no `len` argument on this primitive. The
  16-byte block size is a compile-time constant. Length clamps
  are a non-issue for a raw block cipher.

### Deviations from briefing

1. **Briefing's "no scalar fallback" policy kept as-is.** AES-NI
   is required at runtime on x86_64; `aes128_has_aesni() == 0`
   on a production target is a hard-stop, not a fall-back
   trigger. This is documented in the file header. **Recommend
   main session codify as a DECISIONS.md entry**, e.g. "AES-NI
   required at runtime; Westmere+ server / Goldmont+ Atom floor
   for x86_64 production; constant-time table-lookup AES would
   violate D032, so no scalar fallback is correct." Precedent:
   parallels the AArch64 "FEAT_AES required by D034, no runtime
   dispatch" stance; making the x86_64 side equally explicit
   closes a loose end.

2. **Byte-order claim in the briefing was too strong.** The
   briefing said "AES-NI's aesenc/aesdec operate on little-
   endian xmm registers; the asm handles the byte-swap on
   load/store without committing the caller to it." In practice
   no byte-swap is required — FIPS 197 §3.3's byte-to-state
   mapping (`state[r][c] == bytes[r+4c]`) already matches both
   AES-NI's xmm byte ordering and AArch64 NEON's 16-byte vector
   ordering. The assembly loads/stores with no byte-swap (same
   as every production AES-NI implementation: Intel whitepaper,
   OpenSSL, libsodium, nettle). Documented explicitly at the
   top of both .S files. Not a functional deviation from the
   spec — only a correction of a briefing description.

3. **Alignment contract relaxed from the briefing.** Briefing
   said `round_keys[176]` "must be 16-byte-aligned." The
   implementation uses `movdqu` (x86_64) and `ld1` / `ldr q`
   (AArch64), none of which require alignment. Rather than
   tighten the code to `movdqa` — which would pass a stricter
   contract along to every future caller — the file header
   drops the alignment requirement. The driver still uses
   `alignas(16)` on its local `rk` buffers, so the more
   restrictive version remains testable on demand.

4. **NIST KAT vector "nist-kat-pt-ff" expected ciphertext
   corrected.** Briefing gave `6e29201190152df4ee058139def610bb`;
   Python `cryptography` (libcrypto backend) and the in-file
   scalar reference both produce `35c315a07d944d46775a95d0f87847edU`
   on the same (key, plaintext) pair. Corrected inline with a
   provenance comment. Same pattern as the sha256_test.c
   "64 x 0xff" briefing correction from round I.

5. **D055 and D056 are vacuous for this primitive.** No `len`
   argument (fixed 16-byte block size = compile-time constant),
   no external-input arithmetic inside the asm (fixed 176-byte
   schedule, hardware state in registers). Both disciplines
   were considered and deliberately not applied because the
   surface they protect does not exist here. Flagged so main
   session knows this is a deliberate no-op, not an oversight.

6. **One `bugprone-easily-swappable-parameters` suppression
   class — three sites total.** clang-tidy flagged `gmul(a,b)`
   (commutative by GF(2^8) field axiom — suppression tight)
   and `ref_encrypt/ref_decrypt(rk, in, out)` (canonical AES

### Reviewer findings — MEDIUM (requires main-session decision)

**AArch64 key-expansion cache-timing on S-box scalar lookups.**
Both the clean-Claude subagent review AND the automated Gemini
review converged on the same finding (high-confidence signal per
CLAUDE.md's "if reviewers agree, it's high-confidence"):
`aes128_expand_key` on AArch64 performs 40 data-dependent
`ldrb [x_sbox, x_idx]` loads where `x_idx` is derived from
secret key bytes. The 256-byte S-box spans 4 cache lines, so
cache-line-granularity observation of the key expansion leaks
information about the key through timing. This is a published
attack class (Bernstein 2005, Osvik–Shamir–Tromer 2006).

The in-file rationale ("the key is already a secret; this is
outside the per-message constant-time requirement") conflates
in-memory secrecy with side-channel exposure. The task brief's
explicit constant-time requirement — "The key schedule's only
branch is the loop counter... not the key" — addresses branches
but not memory access patterns.

**Per-message** encrypt/decrypt IS constant-time on both ISAs
(hardware aese/aesd/aesenc/etc. are data-independent by ISA
spec). This finding is scoped strictly to `aes128_expand_key`
on AArch64.

**Options for a follow-up dispatch** (not attempted in this
side session; the retrofit is non-trivial and deserves its own
scoped discussion):

- (a) **NEON `tbl` with S-box in vector registers.** Load the
      256-byte S-box into 16 V-registers (Q0..Q15, 16 bytes
      each) and use a sequence of `tbl` + mask + `orr` to look
      up any byte in constant time. Adds 16 `ld1` at function
      entry and a handful of ops per lookup.
- (b) **`aese` with zero key as a SubBytes oracle.** Place the
      4-byte temp word into a vector lane, zero the other 12
      bytes (so ShiftRows is a no-op on those lanes), `aese v,
      zero_v` applies SubBytes+ShiftRows+XOR-0; extract the
      low 4 bytes back. A known trick documented in the Cortex-A
      optimization guide. Simpler than (a), no table at all.
- (c) **Accept the current implementation** with explicit
      threat-model documentation: "`aes128_expand_key` on
      AArch64 is NOT constant-time on the key-expansion step;
      callers whose threat model includes an adversary capable
      of cache-line-granularity timing observations on the
      expand_key call must rotate keys through a trusted
      helper or migrate to a hardened path." Would require
      a DECISIONS.md entry pinning the stance.

**Recommendation:** path (b) is the lowest-cost fix (no table
loads, minimal register pressure, proven-clean in published
optimization guides). Path (c) is acceptable if the use pattern
is "expand once per long-lived key" — the per-use exposure is
low and most callers likely qualify. Path (a) is heavy and
unlikely to be worth the register pressure.

Not implemented here to respect the side-session file-scope
discipline (the briefing's task is the primitive; a constant-
time hardening pass is a new task). Main session's call.

### Reviewer disagreement — [HIGH] false positive on x86_64 key expansion

The commit's own post-commit Gemini run raised a **[HIGH]**
claim that the `AES128_KEY_ASSIST` macro's
`movdqa xmm3, xmm1; pslldq xmm3, 4` pattern "breaks the
prefix-XOR chain" because `xmm3` is reloaded from `xmm1`
before each shift rather than being shifted cumulatively.
**This is a false positive.** Evidence:

1. **Algebraic equivalence.** Intel AES-NI whitepaper
   Figure 24 uses the AVX three-operand form
   `vpslldq xmm3, xmm1, 0x4` which reads the CURRENT
   (post-XOR) xmm1 and writes the shifted value to xmm3.
   The SSE equivalent is exactly
   `movdqa xmm3, xmm1; pslldq xmm3, 4`. The cumulative
   effect comes from `xmm1` being modified by each `pxor`,
   not from `xmm3` being preserved. Both forms produce an
   xmm3 that equals `(current xmm1) << 4` at each step.

2. **Direct key-schedule cross-check in the test driver.**
   `check_sweep` in `aes128_test.c` runs, for each of 8
   pseudo-random keys, both the asm and the in-file scalar
   reference (FIPS 197 §5.2 per-word expansion, entirely
   independent of the AES-NI macro) and `memcmp`s the
   176-byte schedules byte-for-byte. A broken macro would
   fail this check on the very first block. All 8 blocks
   pass on every cell (native, Denverton, -cpu max).

3. **FIPS 197 Appendix C.1 canonical vector.** `fips-197-c1`
   (the definitive AES-128 test: key = `000102...0f`,
   plaintext = `001122...ff`, ciphertext =
   `69c4e0d8...5a`) passes on all three x86_64 cells. This
   vector exercises the full key schedule — a wrong schedule
   would produce a wrong ciphertext, full stop.

The earlier clean-Claude subagent review and the
pre-commit Gemini review did NOT flag the macro; only the
post-commit Gemini run did. Three-out-of-four reviewers
implicitly confirm correctness; one reviewer disagreed based
on a misreading of SSE-vs-AVX form equivalence. Code left
as-is; recording the disagreement so the next reader doesn't
have to re-derive the same analysis.

### Reviewer findings — LOW (accepted as-is)

- **Register-scrubbing on return (Gemini run 2, LOW).**
  `xmm1`/`v1` (round keys) and `w5-w8` (aarch64 key-expansion
  running words) are not explicitly zeroed before the
  functions return. Defense-in-depth beyond FIPS 197; most
  production AES implementations (OpenSSL, libsodium) rely
  on OS process isolation rather than per-function register
  scrubbing. Accepted; main session may codify a project-
  wide "scrub registers holding key material" D-entry if the
  threat model demands it.
- **Pipeline stall hints (Gemini runs 2+3, LOW).** Round
  keys are loaded immediately before the AES instruction
  that consumes them, which induces load-to-use latency on
  high-performance cores (Neoverse V1/V2, Skylake+). Current
  sequence is clean, correct, and minimal; software
  pipelining (pre-loading rk[i+1] while rk[i] is executing)
  is a worthwhile optimization once mode-amortized callers
  exist and benchmarks justify the scheduling complexity.
- **SSE/AES-NI feature-bit layering (Gemini run 2, LOW).**
  `aes128_has_aesni` doesn't explicitly probe SSE2 feature
  bits before using SSE2 idioms. Accepted: SSE2 is a
  mandatory baseline in the x86_64 ABI (not optional as on
  IA-32), so every x86_64 CPU supports it by construction.
  Adding the redundant check would be noise.
- **Aarch64 FEAT_AES runtime probe (Gemini run 2, LOW).**
  No runtime FEAT_AES probe on AArch64. Accepted per D034:
  FEAT_AES is a baseline for every production aarch64
  profile (Cortex-A76+, Neoverse N1/V1/V2, Apple cores);
  pre-A76 cores are explicitly out of scope.
- **`movdqa` inside KEY_ASSIST macro (Gemini run 3, LOW).**
  The two register-to-register `movdqa` instructions inside
  the macro don't touch memory, so alignment is not a
  correctness concern. The speculative hazard Gemini
  describes ("if future edits introduce memory operands")
  doesn't apply to the current code.
- **pytest success-token coupling (Gemini run 3, LOW).**
  `_assert_pass` matches on the exact string
  `"PASS  all AES-128 checks passed"`. Same pattern as
  `test_sha256.py`'s success-token check — intentional
  precedent from round I. Switching to TAP/JUnit format is
  a cross-cutting tooling change that belongs to main
  session, not this per-primitive wrapper.



- **9 × aesimc per decrypt call.** Deliberate tradeoff
  documented in both .S headers: trades per-call cost for a
  shared encrypt/decrypt key schedule and an unmutated caller
  buffer. Main session may add an `aes128_expand_key_for_decrypt`
  once mode-amortized use (CBC, CTR, GCM) lands.
- **`movdqu` vs `movdqa` fail-fast.** Reviewer disagreement:
  clean-Claude preferred dropping the alignment claim (done);
  Gemini preferred tightening to `movdqa` for fail-fast.
  Canonicalising unaligned eases every caller's life and costs
  one decoded uop per round key on modern x86_64. Chose the
  less-demanding-contract direction.
