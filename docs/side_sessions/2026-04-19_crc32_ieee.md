## Task

Implement a shared-across-arches CRC-32 primitive suitable for
Ethernet FCS computation, in assembly for both supported archs.

### Function contract (both arches)

Host-callable leaf routine following the platform ABI (SysV AMD64 on
x86_64; AAPCS64 on AArch64), so a C driver can link against it for
unit testing.

```c
/* uint32_t crc32_ieee_802_3(const void *data, size_t len); */
```

- **Polynomial:** `0xEDB88320` (reflected form of `0x04C11DB7`).
- **Initial value:** `0xFFFFFFFF`.
- **Final XOR:** `0xFFFFFFFF`.
- **Byte order:** input read as a byte stream; output is the 32-bit
  value per the IEEE 802.3 Annex F.3 reference algorithm. (The
  `FCS_on_the_wire = bswap32(crc)` conversion, if ever needed, is the
  caller's concern — keep this primitive polynomial-correct, not
  wire-format-correct.)
- **Zero-length input:** return `0x00000000` (i.e., `0xFFFFFFFF ^
  0xFFFFFFFF`).

### Implementation targets

- **x86_64:** use the `crc32` instruction (SSE 4.2), which computes
  CRC-32C (Castagnoli) — **wrong polynomial**. So we want
  **PCLMULQDQ** (Intel's "Fast CRC Computation Using PCLMULQDQ
  Instruction" white paper) for the IEEE 802.3 polynomial, with a
  slicing-by-8 table-driven fallback when PCLMULQDQ isn't available.
  D034's ISA feature flags carry `PCLMULQDQ` per profile; respect
  them.
- **AArch64:** use the `crc32*` instructions from the ARMv8 CRC
  extension (`FEAT_CRC32`). These DO implement the IEEE polynomial
  natively — `crc32b`, `crc32h`, `crc32w`, `crc32x` for 1/2/4/8-byte
  updates. D034's AArch64 profiles carry `crc32` as a required feature
  for Cortex-A72 and later. No fallback needed for our target hardware,
  but a build-time assertion that the feature is available is wise.

### Test vectors (IEEE 802.3 and RFC 3720 appendix vectors)

Use these in the host-side C driver:

| Input (ASCII or hex) | Expected CRC-32 (IEEE 802.3) |
|----------------------|------------------------------|
| `""` (empty)          | `0x00000000` |
| `"a"`                 | `0xE8B7BE43` |
| `"abc"`               | `0x352441C2` |
| `"message digest"`    | `0x20159D7F` |
| `"abcdefghijklmnopqrstuvwxyz"` | `0x4C2750BD` |
| 1024-byte block of `0x00` | `0x9A0440DA` |
| 1024-byte block of `0xFF` | `0x5AFF1B50` |

Note: these are the "polynomial correct" values, NOT the on-the-wire
FCS (which is the complement of this in older spec text, or reversed,
depending on reference). Our contract is polynomial-correct; the L2
TX path will bswap/complement as needed.
