## Task

Implement a shared-across-arches SHA-256 primitive conforming to
FIPS 180-4, in assembly for both supported arches, with a host-side
C test driver checking published vectors.

### Function contract (both arches)

Host-callable leaf routine following the platform ABI (SysV AMD64 on
x86_64; AAPCS64 on AArch64), so a C driver can link against it for
unit testing.

```c
/* void sha256(const void *data, size_t len, uint8_t digest[32]);
 *
 * One-shot digest: reads `len` bytes from `data`, writes the
 * 32-byte SHA-256 digest to `digest` in big-endian byte order
 * as FIPS 180-4 prescribes. No streaming API in MVP â€” TLS
 * usage is all one-shot transcript-hash or HKDF-block-sized.
 */
```

- **Byte order of output:** big-endian (matches what RFC 6234 test
  vectors print and what HKDF / TLS 1.3 consumers expect).
- **Length encoding:** 64-bit big-endian bit-length, FIPS 180-4 Â§5.1.1.
- **Empty input:** must return the well-known
  `e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855`.

### Implementation targets

- **x86_64:** use the Intel SHA extensions (`SHA-NI`):
  `sha256rnds2`, `sha256msg1`, `sha256msg2`. These compute two
  rounds per instruction and fold the message schedule directly.
  The round constants K[0..63] live in rodata. D034 carries
  `sha_ni` as an optional profile flag; add runtime CPUID
  dispatch (CPUID leaf 7 ECX bit 29) with a plain-scalar
  software fallback. Keep the fallback on the same entry point
  so tests can exercise both paths.
- **AArch64:** use the ARMv8 SHA-2 extensions (`FEAT_SHA256`):
  `sha256h`, `sha256h2`, `sha256su0`, `sha256su1`. These compute
  four rounds per two-instruction pair. D034 carries `sha2` as
  a required feature for Cortex-A72 and later; a build-time
  assertion is sufficient (no runtime dispatch needed on our
  target profiles).

For both arches: constant-time by construction. SHA-256 has no
secret-dependent branches or memory accesses; the hardware
instructions preserve this and the scalar fallback must also.

**Implementation-side notes (not deviations from briefing):**

- **Dispatcher pattern.** `sha256` is a single `jmp [sha256_ptr]`
  to a function-pointer cache; first call lands on `sha256_resolve`
  which runs CPUID and installs either `sha256_shani` or
  `sha256_scalar` into the cache atomically. Subsequent calls are
  a single indirect jump. This mirrors the CRC-32 dispatcher
  exactly (same `_ptr` + resolver shape in
  `arch/x86_64/crypto/crc32_ieee.S`). Concurrency is safe only
  because writes are aligned 8-byte stores and both write the
  same resolved target; flagging this if the project ever goes
  multithreaded and MVP single-thread (D003) stops holding.

### External-input discipline (main session, when TLS/HTTP lands)

Captured from the post-commit Gemini + security discussion with
Ed. None of the findings on this side session are reachable
attacks on the SHA-256 primitive itself (2^61 overflow needs 2 EiB
of deliverable data; x86_64 r11d truncation is constant-valued;
aarch64 byte-by-byte residue copy is bounded and timing only
leaks public length metadata). But the DISCIPLINE is:

1. **Hash-input length ceiling at the external-input boundary.**
   When the public crypto header lands, define a project-wide
   upper bound (candidate: `FIREASMSERVER_MAX_HASH_LEN = 1 << 40`,
   one TiB â€” many orders of magnitude above any realistic TLS /
   HTTP / filesystem use, many orders below the 2^61 overflow
   horizon). Runtime check lives in CALLERS at the trust
   boundary, not in the primitive itself â€” the primitive's
   trusted-`len` contract matches every C-callable hash API
   (OpenSSL / libsodium / nettle / libgcrypt).

2. **TLS-stack first-principle when it lands.** Hash calls take
   `(data, len)` where `len` has ALREADY been clamped at the
   record-size gate (TLS 1.3: 2^14 plaintext, 2^14+256 cipher-
   text). Make this a stated invariant in the TLS layer's design
   doc, not a runtime check re-derived per call site.
