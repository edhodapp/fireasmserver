## Task

Implement a shared-across-arches SHA-256 primitive conforming to
FIPS 180-4, in assembly for both supported arches, with a host-side
C test driver checking published vectors.

### Function contract (both arches)

Host-callable leaf routine following the platform ABI (SysV AMD64 on
x86_64; AAPCS64 on AArch64), so a C driver can link against it for
unit testing.

```c
/* void sha256(const void *data, size_t len, uint8_t digest[32]);
 *
 * One-shot digest: reads `len` bytes from `data`, writes the
 * 32-byte SHA-256 digest to `digest` in big-endian byte order
 * as FIPS 180-4 prescribes. No streaming API in MVP — TLS
 * usage is all one-shot transcript-hash or HKDF-block-sized.
 */
```

- **Byte order of output:** big-endian (matches what RFC 6234 test
  vectors print and what HKDF / TLS 1.3 consumers expect).
- **Length encoding:** 64-bit big-endian bit-length, FIPS 180-4 §5.1.1.
- **Empty input:** must return the well-known
  `e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855`.

### Implementation targets

- **x86_64:** use the Intel SHA extensions (`SHA-NI`):
  `sha256rnds2`, `sha256msg1`, `sha256msg2`. These compute two
  rounds per instruction and fold the message schedule directly.
  The round constants K[0..63] live in rodata. D034 carries
  `sha_ni` as an optional profile flag; add runtime CPUID
  dispatch (CPUID leaf 7 ECX bit 29) with a plain-scalar
  software fallback. Keep the fallback on the same entry point
  so tests can exercise both paths.
- **AArch64:** use the ARMv8 SHA-2 extensions (`FEAT_SHA256`):
  `sha256h`, `sha256h2`, `sha256su0`, `sha256su1`. These compute
  four rounds per two-instruction pair. D034 carries `sha2` as
  a required feature for Cortex-A72 and later; a build-time
  assertion is sufficient (no runtime dispatch needed on our
  target profiles).

For both arches: constant-time by construction. SHA-256 has no
secret-dependent branches or memory accesses; the hardware
instructions preserve this and the scalar fallback must also.

### Test vectors (FIPS 180-4 + RFC 6234)

Use these in the host-side C driver:

| Input | Length | Expected SHA-256 digest (hex) |
|-------|--------|-------------------------------|
| `""` (empty) | 0 B | `e3b0c44298fc1c149afbf4c8996fb92427ae41e4649b934ca495991b7852b855` |
| `"abc"` | 3 B | `ba7816bf8f01cfea414140de5dae2223b00361a396177a9cb410ff61f20015ad` |
| `"abcdbcdecdefdefgefghfghighijhijkijkljklmklmnlmnomnopnopq"` | 56 B | `248d6a61d20638b8e5c026930c3e6039a33ce45964ff2167f6ecedd419db06c1` |
| `"abcdefghbcdefghicdefghijdefghijkefghijklfghijklmghijklmnhijklmnoijklmnopjklmnopqklmnopqrlmnopqrsmnopqrstnopqrstu"` | 112 B | `cf5b16a778af8380036ce59e7b0492370b249b11e8f07a51afac45037afee9d1` |
| 1 000 000 × `'a'` | 1 MB | `cdc76e5c9914fb9281a1c7e284d73e67f1809a48a497200e046d39ccc7112cd0` |
| 1 × 64-byte block of `0x00` | 64 B | `f5a5fd42d16a20302798ef6ed309979b43003d2320d9f0e8ea9831a92759fb4b` |
| 1 × 64-byte block of `0xff` | 64 B | `af9613760f72635fbdb44a5a0a63c39f12af30f950a6ee5c971be188e89c4051` |

The first five are the FIPS 180-4 + RFC 6234 canonical vectors.
The last two exercise the boundary case of a single full message
block with exactly the padding overflow into a second block.

For defense-in-depth, the C driver should also cross-check by
re-hashing each vector with a second independent scalar-reference
implementation it carries internally (same pattern the CRC-32
driver uses). No external OpenSSL / libcrypto linkage.
